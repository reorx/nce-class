# NCE Class · Milestone 1 PRD / 设计文档

新概念英语（New Concept English）课堂教学辅助系统。本文档为 Milestone 1 的产品需求与界面设计规格，供后续实现使用。

## 1. 概述与目标

NCE Class 帮助英语老师在课堂上对学生分组、即时加减分（"加星"机制），并记录每个学生的背书与作业完成情况；课后向家长推送本堂课的回顾（recap）。系统沉淀每个学生的纵向成长档案，用于持续追踪与日后成绩评定。

系统分两端：
- **老师端**：桌面浏览器为主（未来考虑手机端）。含管理界面与课堂界面。课堂界面需同时满足"老师快速操作"与"投屏给全班看"。
- **学生端**：微信内 H5，家长使用。含加入班级（onboarding）与接收课堂 recap。

## 2. 范围（Milestone 1）

**纳入 M1：**
- 老师端 · 管理界面：班级 / 学生 / 分组方案 / 家长邀请 / 上课记录。
- 老师端 · 课前配置：选班 → 本节课信息 → 选分组方案 → 开始课堂。
- 老师端 · 课堂界面：分组看板、加减分、背书检查、作业检查、调组、撤销、结束课堂生成 recap。
- 老师端 · 学生成长档案（持续追踪）。
- 学生端 H5：加入班级 onboarding；课堂 recap 查看。

**不在 M1（留待 M2 及以后）：**
- 作业系统：老师布置作业、推送给学生、学生完成并上传。
- 微信主动推送（服务号订阅通知 / 小程序订阅消息）。M1 recap 通过链接转发到微信群分发。
- 成绩自动评定算法（M1 只保证数据完整留存）。
- 手机端老师界面。

## 3. 角色与术语

| 术语 | 含义 |
|------|------|
| 学校（School） | 整个教学系统实例，M1 为单一学校 |
| 老师（Teacher） | 需登录；多老师共享同校的班级与学生 |
| 班级（Class） | 隶属学校，含一批学生与若干分组方案 |
| 学生（Student） | 隶属班级；有名字、照片(可选)；来源 `source` = parent(家长自助加入) / teacher(老师手动添加) |
| 家长 | 学生端使用者；M1 不需登录，凭学生专属链接查看 recap |
| 分组方案（GroupTemplate） | 班级下可复用的分组（如"默认分组""周五分组"） |
| 课堂（ClassSession） | 一次"开始课堂"产生的会话；含日期、课次、分组快照、计分 |
| 加星 / 计分 | 以 ±1 为单位的奖惩，计入行为事件流 |
| recap | 一堂课结束后生成的回顾，供家长查看 |

## 4. 技术架构

- **前后端分离**：
  - 老师端：React + Vite + TypeScript（桌面 Web 应用）。
  - 学生端：React H5（微信内打开，移动端布局）。
  - 后端：独立 Node 服务（建议 Hono 或 NestJS）+ TypeScript。
  - 数据库：PostgreSQL（建议 Prisma 作 ORM）。
  - 对象存储：S3 兼容（学生照片）。本地开发可用 MinIO。
- **认证**：
  - 老师：账号登录（建议 Better Auth，邮箱+密码起步）。多老师·单校，校内数据共享，M1 不做细粒度权限。
  - 家长：无需登录。每个学生有不可猜测的 `recapToken`，家长凭学生专属链接查看该生 recap（含历史各堂回顾，详见 §7.5）。家长侧不提供老师端的成长档案（§7.4 为老师专用）。
- **部署**：M1 先本地 / 容器开发，部署后置（后续若要微信内低延迟，倾向国内云）。

> 以上框架为建议默认值，实现时可按团队偏好调整，但需保持"前后端分离 + 关系型数据库 + 事件流计分"的核心结构。

## 5. 数据模型（核心）

计分采用**事件流（ledger）**为单一事实来源，所有派生分数从事件计算。

```
School(id, name)
Teacher(id, schoolId, name, email, passwordHash)
Class(id, schoolId, name)
Student(id, classId, name, photoUrl?, source[parent|teacher], recapToken, createdAt)

GroupTemplate(id, classId, name, isDefault)
TemplateGroup(id, templateId, name, emoji, orderIndex)
TemplateMembership(id, templateGroupId, studentId)

ClassSession(id, classId, teacherId, date, lessonNumber?, lessonTitle?, status[ongoing|ended], startedAt, endedAt)
SessionGroup(id, sessionId, name, emoji, orderIndex)   // 开课时由模板快照生成，可课中调整
SessionMembership(id, sessionId, studentId, sessionGroupId?)  // sessionGroupId 为空 = 未分组/缺席，本节不计分

ScoreEvent(id, sessionId, targetType[group|student], targetId,
           sessionGroupId,   // 学生事件记录"当时所在组"，组事件记录该组
           delta,            // +1 或 -1
           createdAt, createdBy)

CheckRecord(id, sessionId, studentId, type[recitation|homework],
            status, updatedAt)   // 见 §8 枚举；缺记录视为"未检查"
```

**派生计算规则：**
- 学生本节**个人分** = 该 session 中 targetType=student 且 targetId=该生 的 Σdelta。
- 小组本节**组分**（嵌套式） = 该组的 group 事件 Σdelta + 所有 `sessionGroupId=该组` 的 student 事件 Σdelta。
  - 即：给学生 +1 同时计入其所在组；组级 +1 只计入组分。
  - 学生事件记录"当时所在组"，因此课中调组不会回溯改写历史组分。
- 学生**累计个人总分**（跨课，用于学生排行 / 成长） = 跨所有 session 的个人分之和。**不含组分**（组分每课重置、组员会变）。
- 小组分**每节课重置**（每堂课是独立比赛）。

## 6. 计分机制（规则细节）

- 加分 / 扣分均以 **±1** 为单位，可连续点击累加。
- 两个计分目标：**小组**（点列头）与**个人**（学生浮窗）。个人加分嵌套计入其组（见 §5）。
- 所有加减分写成 `ScoreEvent`，满足"记录行为"诉求；recap 的"亮眼 / 被提醒"名单从事件流派生，规则如下（阈值可配置）：
  - **被提醒** = 本节存在任意负向事件（任一 −1）的学生。
  - **亮眼** = 本节个人净加分 ≥ 2 的学生（默认阈值 2；若希望"取前 N 名"亦可配置，但 M1 默认用固定阈值，保证确定性）。
- **撤销**：撤销最近一次 ScoreEvent（课堂界面 dock 提供）。
- 背书 / 作业为**定性标签**，本身不直接加减分；老师若要因其表现奖惩，仍走独立的 ±1。数据完整留存，供日后成绩评定纳入考量。
- 缺席 / 未分组（SessionMembership.sessionGroupId 为空）的学生本节不参与计分。

## 7. 功能与界面

### 7.1 老师端 · 管理界面

**首页 = 班级列表**
- 顶部：学校名 + 导航（`班级` / `老师`）+ 当前老师菜单。
- 班级卡片：班级名、学生数、上次上课时间、`▶ 开始上课`、`管理`。
- `+ 新建班级`。
- `老师` 导航：多老师·单校的老师管理（M1 可极简：列表 + 邀请老师）。

**班级详情**（子标签）
- `学生`：头像 + 名字 + 累计总分的名单网格；点学生 → 成长档案。学生有两个来源：家长经班级邀请链接**自助加入**（`source=parent`，填完照片+名字即为正式学生），或老师 `手动添加`（`source=teacher`，可仅填名字、照片占位，老师可后补照片）。因班级通用链接不做认领匹配，可能出现重复学生，老师可在名单里**删除 / 合并**重复项。
- `分组方案`：管理默认 / 其他分组方案。
- `邀请家长`：邀请链接 + 二维码 + 复制；家长微信打开完善信息即加入。
- `上课记录`：历次 ClassSession 列表，可查看对应 recap。

### 7.2 老师端 · 课前配置（开始课堂）

- 顶部：返回班级列表 + 班级名 + 学生数。
- **本节课**（可选）：课次号 + 课题，进入 recap 与档案。
- **分组方案**：选已存方案（默认 / 其他）或 `+ 新建分组`；下方预览各组成员，可拖拽微调。
- **未分组 / 今日缺席**暂存区：缺席学生拖入；留在此处则本节不计分。
- `开始课堂 →` 进入课堂界面（按当前分组生成 SessionGroup 快照）。

### 7.3 老师端 · 课堂界面（核心，投屏）

整体：**看板列**（每组一列，列头含组名/emoji + 组分）+ 顶部信息条（班级、课次、计时）+ 底部 **dock**。dock 含 **view 切换**：`上课`（默认）/ `背书检查` / `作业检查` / `调组`，以及 `撤销`、`结束课堂`。四个 view 共用同一套数据，仅呈现/交互不同——一致性强。

**上课 view**
- 每组一列，列内学生卡（头像 + 名字 + 本节个人分）；卡上两个状态 label：`背`（背书）/ `作`（作业），按状态着色（统一配色见 §8：绿/黄/蓝/红，灰=未检查）。
- 点**学生卡** → 屏幕**居中浮窗**：
  - 加分区：等宽大按钮 `−1`（红）/ `+1⭐`（绿），可连点，实时显示本次累计（提示"个人+N，小组同步+N"）。
  - `背书`区：已背完 / 背完部分 / 没背（当前高亮）。
  - `作业`区：优秀 / 完成 / 没交（当前高亮）。
  - 浮窗**停留**：加分连点、背书/作业选完高亮，点"完成"或点窗外关闭。
- 点**列头** → 给整组 +1⭐（快捷）；列头亦提供 `−`（组级 −1）。组级加减分对应 `ScoreEvent(targetType=group)`，只计入组分、不分摊到个人。

**背书检查 view / 作业检查 view**
- 切换为"按状态分段"的视角（非看板、非按组）：有几个状态就分几段。
  - 背书：`未检查` / `已背完` / `背完部分` / `没背`。
  - 作业：`未批改` / `优秀` / `完成` / `没交`。
- 每段：标题（配色圆点）+ 人数；下方学生以**带头像的 badge** 横向流式排列（与课堂卡片风格一致），badge 按所在段着色，保留小组号小标签做参考。
- 点 badge → 居中浮窗选状态 → badge 归入对应段。顶部显示总进度（如"已检查 11/16"）。
- 作业检查可选**盖章式批改**：先选一个章（如"完成"），再连点学生快速盖（大批同状态时提速）。此为可延后的 nice-to-have，非 M1 关键路径。

**调组**
- 拖拽学生卡在组间移动；课中可随时调整。调组只影响后续事件归属，不回溯历史组分。

**结束课堂**
- 结束 session（status=ended），生成本堂 recap。

### 7.4 老师端 · 学生成长档案（持续追踪）

从班级名单点入某学生：
- 顶部：头像 + 名字 + 已上课数；累计指标 `个人总分` / `累计加星` / `累计扣分`。三者均**只统计个人事件**（targetType=student），不含组级事件，与 `个人总分` 口径一致，避免重复计入组分。
- `个人分趋势`：近若干节柱状走势。
- `每节课明细`表：日期/课次 | 所在组（组分）| 个人分 | 作业状态 | 背书状态。

### 7.5 学生端 H5（微信内，家长用）

**加入班级（onboarding）**
- 家长点**班级通用邀请链接**在微信打开：展示"老师邀请你加入 + 班级 / 学校"。
- 上传学生照片（圆形，课堂展示用）+ 填学生名字 → `确认加入班级`，**在该班创建一个正式学生**（`source=parent`，附带其 `recapToken`）。无认领匹配；重复由老师在管理端删除/合并（见 §7.1）。
- 加入成功后页面**跳转到该学生的专属链接**（即其 recap/档案页），提示家长**收藏此页**——以后每堂课后回到这里即可看回顾。

**课堂 recap（学生专属链接）**
- 访问凭据：每个学生的 `recapToken` 专属链接，**一户一链接**，承载该生的个性化内容。M1 **不做群内公共版**（一条群链接无法做每户不同的个性化、且会泄露他人信息）。老师口头提醒家长课后到自己孩子的链接查看。
- 页面默认展示**最新一堂课的 recap**，并可下翻查看历史各堂。
- 内容：
  - 顶部：班级 + 课次 + 日期。
  - **本人个性化卡**：所在组、本节个人分、作业状态、背书状态。
  - `🏆 各组得分`：排名，本人所在组高亮标注。
  - `🌟 表现亮眼`：名字 badge（派生规则见 §6）。
  - `⚠️ 被老师提醒`：名字 badge（派生规则见 §6）。

## 8. 状态枚举

- **作业（homework）**：`优秀` / `完成` / `没交`；默认 `未检查`。
- **背书（recitation）**：`已背完` / `背完部分` / `没背`；默认 `未检查`。
- 两者都存于 `CheckRecord.status`；"未检查" = 该 session 该生该类型**无记录**（非独立枚举值）。作业检查 view 里分段标题写作"未批改"，与此处"未检查"是**同一状态**，仅措辞不同。
- 课堂卡片角标与检查 view 配色：绿（已背完/优秀）/ 黄（背完部分）/ 蓝（完成）/ 红（没背/没交）/ 灰（未检查）。

## 9. 隐私与边界

- **"被老师提醒"名单**：每户在自己孩子的 recap 链接里都能看到完整的班级 recap（含"被提醒"名单），故该名单 M1 **对全班家长可见**（按用户决定）。**标注为可调**——后续可改为"仅本人家长可见"（届时被提醒部分只在该生自己的链接渲染）。
- 学生照片仅用于课堂识别与 recap 展示。
- `recapToken` 为不可猜测的长随机串，作为家长无登录访问的 bearer 凭据；泄露即等于该生 recap 可被访问，故不应包含敏感信息。

## 10. 假设与开放问题

- 假设典型班级约 10–20 人、3–5 组（界面按此密度设计，应自适应更多/更少）。
- "本节课"信息为可选；若不填，recap/档案以日期标识。
- 作业检查针对"上一堂课布置的作业"；M1 仅按 session 记录，不与具体作业实体关联（作业实体属 M2）。
- 老师端权限：M1 校内老师等同可见可操作，暂不做角色细分。
- 待 M2：作业布置与提交、微信主动推送、成绩自动评定、手机端老师界面。

## 11. 实现分阶段建议

本 PRD 覆盖老师管理、四视图课堂、成长档案、学生端 H5、认证与存储——作为一个 milestone 合理，但作为单个实现计划偏大。建议下游实现计划按以下顺序分阶段，每阶段可独立验证：

1. **基础**：数据模型 + 迁移 + 老师认证（多老师·单校）+ 对象存储接入。
2. **老师管理端**：班级 / 学生（含通用邀请链接、删除合并）/ 分组方案 / 班级详情。
3. **课堂引擎**：开始课堂（分组快照）→ 四视图（上课/背书检查/作业检查/调组）→ 事件流计分 → 撤销 → 结束课堂。
4. **派生与档案**：组分/个人分/累计总分计算、recap 生成、学生成长档案。
5. **学生端 H5**：onboarding 自助加入 + 学生专属链接 recap。

（盖章式批改、缺席暂存等可在对应阶段后期补充。）
